1. Spring框架

框架（Framework）：在Java开发领域中，具体的表现为一系列的jar文件，它通常约定了一些特殊的数据处理流程或编程方式，以解决特定的问题。

Spring框架主要解决的问题是：创建对象，并管理对象。简单的说，在传统开发模式下，当需要某个对象时，可以在项目中编写例如User user = new User();这样的代码来创建对象，也可以通过user.setName("Alex");这样的方式来为其中的属性赋值，或者将其声明为public static User user;以添加static修饰符使之在内存中的存在时间更长等等，当使用Spring框架后，就不必再写以上这些代码了，相关的工作就可以交给Spring框架来完成，主要是通过配置文件或注解来解决这些问题。

2. 创建基于Spring框架的项目

创建Maven Project，在创建过程中，勾选Create a simple Project，Group Id为cn.tedu，Artifact Id为spring01，Packaging选择jar即可。

创建完成之后，Spring项目需要在pom.xml中添加spring-context的依赖，关于依赖的代码，可以在https://mvnrepository.com/搜索该依赖的名称，确定所使用的依赖及版本，然后，在pom.xml中手动添加<dependencies>节点，粘贴从网站上搜索到的依赖的代码：

x
  <dependencies>

    <dependency>
      <groupId>org.springframework</groupId>
      <artifactId>spring-context</artifactId>
      <version>5.1.5.RELEASE</version>
    </dependency>

  </dependencies>

3. 通过Spring框架获取对象

3.1. 通过无参数构造方法创建对象

先从http://doc.canglaoshi.org/config/spring-context.zip下载配置文件的压缩包，解压后得到applicationContext.xml文件，将该文件复制到项目的src/main/resources文件夹之下，这个文件就是Spring框架的配置文件。

假设需要通过Spring创建一个日期类型的对象，可以在配置文件中添加：

x
  <!-- 通过Spring框架创建Date对象 -->
  <!-- id属性：自定义名称，表示Bean Id，后续将通过这个值获取对象，通常推荐使用类名将首字母改为小写 -->
  <!-- class属性：需要创建哪个类的对象，取值是类的全名 -->
  <bean id="date" class="java.util.Date"></bean>

添加完成后，如果启动Spring，Spring框架就会自动创建java.util.Date类的对象，并且对象名为date，开发人员就可以通过date这个名称获取到对象了！

可以自定义一个带main()方法的类，加载以上配置文件，就可以得到Spring容器，并从容器中获取所需的对象：

x
public class SpringTest {
​
  public static void main(String[] args) {
    // 加载Spring配置文件，获取Spring容器
    ClassPathXmlApplicationContext ac
      = new ClassPathXmlApplicationContext("applicationContext.xml");
    
    // 从Spring容器中获取对象，调用getBean()方法时，参数就是配置文件中的Bean Id
    Date date = (Date) ac.getBean("date");
    
    // 测试
    System.out.println(date);
    
    // 关闭
    ac.close();
  }
​
}

3.2. 通过静态工厂方法创建对象

静态工厂方法：某个类中存在方法，该方法是使用static修饰的，并且，该方法的返回值类型就是当前类的类型，例如Calendar类就存在静态工厂方法，该类中存在：

xxxxxxxxxx
public static Calendar getInstance() {
  // ...
}

由于Calendar是一个抽象类，不可能通过new Calendar()这样的语法创建对象，所以，需要创建对象，就通过以上静态工厂方法，在Spring的配置文件中：

x
  <!-- 3.2. 通过静态工厂方法创建对象 -->
  <!-- factory-method：工厂方法的名称 -->
  <bean id="calendar" class="java.util.Calendar" 
    factory-method="getInstance"></bean>

3.3. 通过实例工厂方法创建对象

如果某个类不能通过无参数构造方法创建对象（根本就没有无参数构造方法，或者是抽象类），也没有静态工厂方法，但是，只要有实例工厂方法，也是可以由Spring创建对象的！

实例工厂方法：另外有一个类中存在方法，可以创建所需要的类的对象！例如需要创建UserDao类，但是，UserDao不满足无参数构造方法和静态工厂方法的前提，却存在UserDaoFactory类，这个类中有某个方法的返回值类型是UserDao，就可以了，这个返回UserDao类型的方法就是实例工厂方法，在实际操作时，需要先创建出UserDaoFactory的对象（实例），然后，再调用创建出来的对象的方法，以得到UserDao类型的对象。

例如：

x
public class UserDao {
​
  public UserDao(Object object) {
  }
  
}
x
public class UserDaoFactory {
  
  public UserDao newInstance() {
    return new UserDao(null);
  }
​
}

在配置时，需要先配置UserDaoFactory，使得Spring能创建出工厂类UserDaoFactory的对象，然后，在配置UserDao时，通过factory-bean以指定工厂：

x
  <!-- 3.3. 通过实例工厂方法创建对象 -->
  <!-- factory-bean：调用哪个Bean的工厂方法，以创建当前类的对象，取值为Bean Id -->
  <bean id="userDaoFactory" class="cn.tedu.spring.UserDaoFactory"></bean>
  <bean id="userDao" class="cn.tedu.spring.UserDao"
    factory-bean="userDaoFactory"
    factory-method="newInstance"></bean>

4. Spring管理的Bean的作用域与生命周期（仅了解）

由Spring管理的Bean，默认是单例的，也就是说：根据同一个Bean Id反复获取，得到的对象会是同一个对象！

可以在<bean>节点中添加scope属性，以配置是否单例，例如：

x
<!-- scope属性：作用域，常用取值可以是singleton或prototype，默认是前者，表示“单例”，后者表示“非单例” -->
<bean id="user" class="cn.tedu.spring.User" scope="prototype"></bean>

在配置时，在单例的前提下，还可以在<bean>节点中添加lazy-init属性，是配置该Bean是否通过懒汉式模式进行管理（懒加载）：

x
<!-- lazy-init属性：是否懒加载，取值可以使用布尔值，默认值为false -->
<bean id="user" class="cn.tedu.spring.User"
    scope="singleton" lazy-init="true"></bean>

由于Bean对象都是由Spring框架进行管理的，具体来说，就是由Spring框架决定何时创建、何时销毁，如果需要在创建过程和销毁之前需要执行特定的操作，就需要有对应的生命周期方法，也就是初始化方法和销毁方法，则Spring会在创建对象之后自动调用初始化方法，在销毁对象之前自动调用销毁方法，开发人员就可以通过这2个方法来决定创建和销毁时需要执行哪些操作。

在需要添加生命周期方法的类中，可以自定义2个方法，分别表示初始化方法和销毁方法，关于方法的声明：

  • 应该使用public权限；
  • 使用void表示返回值类型；
  • 方法名称可以自定义；
  • 参数列表为空。

所以，可以自行添加2个方法，例如：

x
public class User {
  
  public User() {
    System.out.println("\tUser.User()");
  }
  
  public void init() {
    System.out.println("\tUser.init()");
  }
  
  public void destroy() {
    System.out.println("\tUser.destroy()");
  }
  
}

然后，在<bean>节点添加属性配置：

x
<!-- init-method属性：生命周期方法的初始化方法的名称 -->
<!-- destroy-method属性：生命周期方法的销毁方法的名称 -->
<bean id="user" class="cn.tedu.spring.User"
    init-method="init" destroy-method="destroy"></bean>

5. Spring IoC

5.1. 基本概念

IoC：Inversion of control，控制反转。在传统开发模式下，程序员自行编写代码创建对象，例如User user = new User();，也可以自行确定对象的属性值，例如user.setName("David");，如果都是程序员自行编写代码来实现，可以理解为“程序员掌控了对象的控制权”，当使用Spring框架后，都是通过Spring的配置来完成，最终是由Spring框架来创建对象，并确认属性的值，可以理解为“程序员将控制权交给了框架”，所以，称之为“控制反转”；

DI：Dependency Injection，依赖注入。假设项目中有UserLoginServlet，其中，处理数据的过程中，可能涉及数据库编程，就会调用另外的UserDao类的功能来实现，这样的话，就可以称之为UserLoginServlet是依赖于UserDao的，通过Spring框架，可以实现自动为UserLoginServlet中的UserDao属性赋值，这个赋值操作就称之为“注入”，由于是存在依赖关系，所以，这种注入也就称之为“依赖注入”。

以上2个概念中，IoC是Spring框架所实现的效果，而DI是Spring框架所使用的做法，完整的说，Spring框架通过DI实现了IoC。

5.2. 通过SET方式为属性注入值

假设存在SampleBean类，类中有String name属性，需要给该属性注入值，首先，就需要添加该属性对应的SET方法（不强制要求有GET方法）：

xxxxxxxxxx
public class SampleBean {
  
  // Frank
  private String name;
​
  public void setName(String name) {
    this.name = name;
  }
  
  public String getName() {
    return name;
  }
​
}

在配置时，为<bean>节点添加子级的<property>节点，以指定属性名和属性值：

x
  <!-- property节点：配置属性的值 -->
  <!-- name属性：被注入的属性的名称 -->
  <!-- value属性：被注入的值 -->
  <bean id="sampleBean" class="cn.tedu.spring.SampleBean">
    <property name="name" value="Frank"></property>
  </bean>

则Spring在创建对象的同时，就会为该属性注入所配置的值！

其实，Spring框架在处理时，读取到以上<property>节点时，发现name="name"，由于该属性的值是"name"，Spring框架会就自行组织出setName这样的名称，作为需要调用的方法名，并且，将value的值作为调用方法的参数！所以，就要求SampleBean这个类中存在名为setName的方法，并且，该方法能够接收"Frank"作为参数！

所以，在使用SET方式注入时，其实，要求<property>节点中name属性的值与类中的SET方法的名称相对应！但是，属性对应的SET方法的命名方式是相对固定的，甚至各开发工具生成SET方法的规则与Spring组织出SET方法的规则是一模一样的，所以，在不纠结这些细节的情况下，可以简单的认为“在<property>节点中配置的name属性的值就是类中的属性名”！

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

附：设计模式之单例模式

如果需要某个类是单例的，例如：

x
public class King {
}

其实，普通类肯定不是单例的，在类的外部，可以随意创建若干个对象：

x
King k1 = new King();
King k2 = new King();
King k3 = new King();

为了避免随意创建对象，会将类的构造方法私有化：

x
public class King {
  private King() {
  }
}

则在类的外部，就不可以通过new King()的语法来创建对象了！

当然，私有化构造方法的目的不是“不允许创建对象”，而是“不允许随意创建对象”，使用了private权限后，在类的内部还是可以创建对象的，例如：

x
public class King {
  private King king = new King();
  private King() {
  }
}

然后，提供一个公有的方法，使得类的外部是可以获取到king属性的：

x
public class King {
  private King king = new King();
  
  private King() {
  }
  
  public King getInstance() {
    return king;
  }
}

所以，在类的外部是不可以通过new King()的语法来创建对象的，但是，通过调用getInstance()方法仍可以获得对象，并且，无论调用多少次该方法，得到的都是同一个对象！

以上代码还存在一个巨大的缺陷：如果想到调用getInstance()方法，必须先有该类的对象，而得到该类的对象的唯一方式就是调用该方法！所以，就存在：没有对象就调用不了方法，调用不了方法就得不到对象。

所以，为了解决以上问题，还应该在getInstance()方法之前添加static进行修饰，就可以实现“在不需要创建对象的情况下调用getInstance()方法”：

xxxxxxxxxx
public class King {
  private King king = new King();
  
  private King() {
  }
  
  public static King getInstance() {
    return king;
  }
}

由于被static修饰的成员不可以直接访问非static 成员，所以，getInstance()方法内部是不允许访问king属性的，为了保证king属性的使用，该属性也需要添加static修饰符：

x
public class King {
  private static King king = new King();
  
  private King() {
  }
  
  public static King getInstance() {
    return king;
  }
}

以上代码是单例模式中的“饿汉式单例模式”的代码，另外，还有“懒汉式单例模式”！

懒汉式单例模式的基础代码是：

x
public class King {
  private static King king = null;
  
  private King() {
  }
  
  public static King getInstance() {
    if (king == null) {
      king = new King();
    }
    return king;
  }
}

以上代码是存在线程安全问题的，所以，需要对以上代码加锁来解决问题：

x
public class King {
  private static King king = null;
  private static final Object lock = new Object();
  
  private King() {
  }
  
  public static King getInstance() {
    synchronized (lock) {
      if (king == null) {
        king = new King();
      }
    }
    return king;
  }
}

当添加了锁之后，线程安全问题就解决了，但是，每次尝试获取对象时，都需要加锁，就会导致getInstance()方法运行效率较低，所以，并不是每次都需要锁，则还应该将代码调为：

x
public class King {
  private static King king = null;
  private static final Object lock = new Object();
  
  private King() {
  }
  
  public static King getInstance() {
    // 判断有没有必要锁
    if (king == null) {
      synchronized (lock) {
        // 判断是否需要创建对象
        if (king == null) {
          king = new King();
        }
      }
    }
    return king;
  }
}
